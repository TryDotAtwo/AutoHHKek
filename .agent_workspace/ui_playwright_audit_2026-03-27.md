# UI Audit 2026-03-27

Проверка выполнена вручную через Playwright MCP на локальном `dashboard` по адресу `http://127.0.0.1:8768`.

## Что было пройдено

- Вкладки: `Агент`, `Вакансии`, `Карточка`, `Ход работы`
- Быстрые действия в чате: `Запустить анализ`, `Открыть hh.ru`, `Открыть вакансии`
- Кнопки onboarding/pipeline на вкладке `Агент`
- Открытие карточки вакансии из колоночного списка
- Действия карточки: `Подходит`, `Сомневаюсь`, `Не подходит`, `Сохранить письмо`, `Собрать план отклика`, `Откликнуться`, `Следующая карточка`, `Следующая вакансия`
- Чат: ручная отправка сообщения `help`
- Модалка деградации LLM: `Продолжить на эвристиках`
- Визуальная проверка desktop и узкой mobile-ширины

## Найденные дефекты

### 1. Массовая битая кодировка в UI

Критичность: высокая

Подтверждено в нескольких местах:

- navigation snapshot показывает битый label вместо `Разделы дашборда`
- тексты в chat-ответах после `help`, `Запустить анализ`, `Открыть hh.ru`
- часть summary/reason на вкладке `Вакансии`
- заголовки некоторых вакансий
- текст сопроводительного письма в `Карточка`
- часть событий на вкладке `Ход работы`

Симптомы:

- строки вида `07@01>BG8:` и `\x1d0CG=K9 ...`
- русские сообщения с управляющими символами и мусором

Это уже не единичный data issue, а системная проблема декодирования/нормализации текста между backend snapshot и frontend.

### 2. Mobile layout фактически сломан

Критичность: высокая

При ширине `390px` интерфейс схлопывается в узкую вертикальную колонку. Контент становится практически нечитаемым:

- табы и контент режутся по ширине
- основная рабочая область и боковой чат не перестраиваются в пригодный mobile layout
- карточки и списки вакансий визуально непригодны для использования

### 3. `Откликнуться` не выполняет сценарий отклика в текущем окружении

Критичность: высокая

После нажатия кнопки статус отклика становится:

- `Failed to start Playwright browser for hh.ru apply flow: [WinError 5] Отказано в доступе`

С точки зрения пользователя кнопка неработоспособна в текущей конфигурации. Это либо дефект запуска браузера/прав доступа, либо приложение не умеет корректно сообщать prerequisite и блокировать кнопку заранее.

### 4. `Открыть hh.ru` отрабатывает неявно и даёт битый текст в журнале

Критичность: средняя

Кнопка отвечает `200`, но в интерфейсе нет ясного подтверждения успешного результата:

- новый tab в Playwright не открылся
- понятного UI-confirmation нет
- в чате появляется битое сообщение

Нужно проверить, какой именно UX ожидается: открытие внешнего браузера, новый tab или статус в activity/log.

### 5. `Запустить анализ` обновляет статус с битой строкой прогресса

Критичность: средняя

Функционально запрос проходит, но пользователь видит сломанный текст вместо нормального прогресса анализа. Это делает статус почти бесполезным.

### 6. В activity/log присутствуют испорченные записи

Критичность: средняя

На вкладке `Ход работы` часть карточек читается нормально, часть содержит мусор вместо русского текста. Это мешает диагностике сценариев и снижает ценность audit trail.

### 7. Сопроводительное письмо в карточке вакансии может открываться уже в испорченном виде

Критичность: средняя

В `Карточка` textarea `Сопроводительное письмо` содержит битый текст ещё до ручного редактирования. Даже если `Сохранить письмо` работает по API, пользователь не может доверять содержимому.

### 8. Отсутствует явная защита от неготовых/недоступных сценариев

Критичность: средняя

Часть действий остаётся доступной, хотя runtime фактически не может их выполнить:

- apply flow падает по `WinError 5`
- hh/LLM зависимые сценарии переходят в degraded mode уже после клика

Желательно заранее показывать блокировку, prerequisite или более ясный статус, а не давать пользователю кликать в заведомо неготовый сценарий.

- Git status не проверялся из-за `dubious ownership`, это не мешало UI-аудиту
